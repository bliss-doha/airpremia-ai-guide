# 05. 좋은 예 vs 나쁜 예

실제 업무 상황에서 어떻게 행동하면 좋은지 사례로 살펴봅니다.

## 시나리오 01 · 고객 응대 메일 초안 작성

**나쁜 예**: "홍길동 고객(010-1234-5678, 예약번호 AB123456)이 환불 요청한 사례 정리해줘"
→ 개인정보가 외부로 전송되어 위수탁 미체결로 법적 문제가 발생할 수 있습니다.
{: .warning }

**좋은 예**: "환불 요청 고객 대응 메일을 정중한 톤으로 초안 작성해줘. {이름}, {예약번호}를 포함한 일반적 사례 기준으로."
→ 실제 식별정보 대신 플레이스홀더({이름}·{예약번호})로 일반 양식만 요청합니다.
{: .tip }

## 시나리오 02 · 운항 매뉴얼 요약

**나쁜 예**: 운항 매뉴얼 PDF를 개인 Gemini에 업로드해 요약 요청
→ 기밀자료가 외부로 전송됩니다. 학습에 활용되어 다른 사용자 답변에 섞일 가능성이 있습니다.
{: .warning }

**좋은 예**: M365 Copilot에서 매뉴얼 SharePoint 파일을 참조해 요약 작성
→ 회사 테넌트 내부에서 처리됩니다. 외부 전송이 없습니다.
{: .tip }

## 시나리오 03 · 코드 디버깅 도움 받기

**나쁜 예**: API Key, DB 접속 정보가 들어있는 코드 그대로 개인 ChatGPT에 붙여넣기
→ 인증정보가 노출됩니다. 노출된 정보는 즉시 폐기·재발급 수행합니다.
{: .warning }

**좋은 예**: API Key 부분은 "[REDACTED]"로 가린 채 Claude Team에 입력해 로직 디버깅 요청
→ 로직만 검토합니다. 인증정보는 입력하지 않습니다.
{: .tip }
